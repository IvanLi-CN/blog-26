# 内容缓存系统实现文档

## 概述

为了提升项目性能，我们实现了一个内容缓存系统，将所有内容（markdown 纯文本内容）缓存到 SQLite 数据库中。该系统在项目启动时初始化，之后每 10 分钟自动刷新一次。

## 功能特性

### 1. 数据库表结构

#### Posts 表（文章缓存）

- 存储博客文章和项目文章
- 包含完整的 frontmatter 数据和 markdown 内容
- 支持内容哈希检测变更
- 字段包括：id, slug, type, title, excerpt, body, publishDate, updateDate, draft, public, category, tags, author, image, metadata, contentHash, lastModified, createdAt, updatedAt

#### Memos 表（闪念缓存）

- 存储闪念内容
- 支持标签和附件
- 字段包括：id, slug, title, body, publishDate, updateDate, public, tags, attachments, contentHash, lastModified, createdAt, updatedAt

### 2. 缓存管理器

#### 核心功能

- **自动刷新**：项目启动时立即执行一次刷新，然后每 10 分钟自动刷新
- **增量更新**：通过内容哈希检测变更，只更新有变化的内容
- **错误处理**：WebDAV 请求失败时不会影响整体系统运行
- **性能优化**：批量操作，减少数据库 I/O

#### 实现文件

- `src/lib/content-cache.ts` - 缓存管理器核心逻辑
- `src/lib/schema.ts` - 数据库表定义
- `src/lib/db.ts` - 数据库初始化和缓存管理器启动

### 3. API 接口

#### tRPC 路由：`contentCache`

**公开接口：**

- `getStatus` - 获取缓存状态统计
- `getPosts` - 获取缓存的文章列表（支持分页和类型过滤）
- `getMemos` - 获取缓存的闪念列表（支持分页）

**管理员接口：**

- `refresh` - 手动触发缓存刷新

### 4. 管理界面

#### 缓存管理页面：`/admin/cache`

- **状态概览**：显示文章和闪念的缓存数量和最后更新时间
- **内容列表**：分页显示缓存的文章和闪念
- **手动刷新**：管理员可以手动触发缓存刷新
- **实时更新**：使用 tRPC 客户端实现实时数据更新

## 技术实现

### 1. 数据流程

```
WebDAV/本地文件 → fetchContent() → 内容解析 → 哈希计算 → 数据库缓存
                                                    ↓
                                            定时刷新（10分钟）
```

### 2. 智能缓存策略

#### 基于修改时间的变更检测

- **WebDAV 文件**：使用文件的 `lastmod` 时间戳检测变更
- **本地文件**：使用内容哈希检测变更
- **智能跳过**：只有当文件修改时间比缓存更新时，才重新处理

#### 缓存刷新策略

- **启动时初始化**：确保系统启动时有最新数据
- **定时刷新**：每 10 分钟检查并更新变更的内容
- **增量更新**：只处理有变化的内容，大幅提高效率
- **错误容忍**：单个文件处理失败不影响整体缓存

#### 性能表现

```
首次刷新：~18秒（处理所有文件）
后续刷新：~1秒（无变更时快速跳过）
缓存命中率：100%（文件无变更时）
```

### 3. 性能优化

- **批量操作**：使用批量插入和更新减少数据库操作
- **内容哈希**：快速检测内容变更，避免不必要的更新
- **分页查询**：API 支持分页，避免一次性加载大量数据
- **异步处理**：缓存刷新在后台异步执行，不阻塞主线程

## 使用方法

### 1. 自动运行

系统会在项目启动时自动初始化并开始工作，无需手动配置。

### 2. 手动管理

访问 `/admin/cache` 页面可以：

- 查看缓存状态
- 浏览缓存内容
- 手动触发刷新

### 3. API 调用

```typescript
// 获取缓存状态
const status = await trpc.contentCache.getStatus.query();

// 获取文章列表
const posts = await trpc.contentCache.getPosts.query({
  limit: 20,
  offset: 0,
  type: 'post'
});

// 手动刷新缓存（需要管理员权限）
await trpc.contentCache.refresh.mutate();
```

### 2. 流控和重试机制

#### WebDAV 请求流控

- **请求间延迟**：每个 WebDAV 请求之间有 100ms 延迟，避免过于频繁的请求
- **智能重试**：遇到 429 错误时自动重试，最多重试 3 次
- **退避策略**：429 错误使用随机退避（5-10秒），其他错误使用固定延迟（1-3秒）

#### 具体实现

```typescript
// 配置参数
const WEBDAV_REQUEST_DELAY = 100; // 请求间延迟
const WEBDAV_RETRY_ATTEMPTS = 3; // 重试次数
const WEBDAV_RETRY_DELAY = 5000; // 重试延迟

// 双层流控机制
1. WebDAV 客户端层面：fetchWithRetry() 函数处理 429 错误和网络错误
2. 内容缓存层面：批量处理时的请求间延迟和整体重试逻辑
```

#### 错误处理策略

- **429 错误（速率限制）**：自动重试，使用指数退避 + 随机延迟
- **网络错误**：自动重试，使用固定延迟
- **其他 HTTP 错误**：记录错误但继续处理其他文件
- **数据库错误**：记录错误，不影响后续操作

## 注意事项

### 1. WebDAV 限制

- 由于 WebDAV 服务器可能有请求频率限制，初次启动时可能会遇到 429 错误
- 系统会继续处理其他文件，失败的文件会在下次刷新时重试

### 2. 数据一致性

- 缓存数据可能会有最多 10 分钟的延迟
- 如需立即更新，可以使用管理界面手动刷新

### 3. 存储空间

- 缓存会占用额外的数据库存储空间
- 建议定期监控数据库大小
